## Executive Summary

- The [Docs] section in both pickers (`<leader>ac` for Claude, `<leader>ao` for OpenCode) is generated by `create_docs_entries()` in `entries.lua`, which scans `{base_dir}/docs/` for `*.md` files (excluding README.md)
- The scan function (`scan.scan_directory`) explicitly excludes README.md files, which is the opposite of what the task requires
- The fix requires modifying `create_docs_entries()` in `entries.lua` to show only `docs/README.md` instead of scanning for all `*.md` files
- Both pickers share the same entry-creation code path, differentiated only by the `config` parameter (base_dir: `.claude` vs `.opencode`)

## Context & Scope

The user wants to limit the [Docs] section in each picker to show only a single file:
- `<leader>ac` (Claude picker): show only `.claude/docs/README.md`
- `<leader>ao` (OpenCode picker): show only `.opencode/docs/README.md`

Currently, the [Docs] section shows all top-level `.md` files in the docs directory (except README.md). In various projects, files like `parity-summary.md` and `STANDARDS_QUICK_REF.md` appear, cluttering the picker.

### scan_directory README Exclusion

**File**: `lua/neotex/plugins/ai/claude/commands/picker/utils/scan.lua` (lines 23-42)

```lua
function M.scan_directory(dir, pattern)
  local files = {}
  local file_paths = vim.fn.glob(dir .. "/" .. pattern, false, true)

  for _, filepath in ipairs(file_paths) do
    local filename = vim.fn.fnamemodify(filepath, ":t")
    local is_readme = filename == "README.md"

    if not is_readme then       -- <-- README.md explicitly excluded
      local name = vim.fn.fnamemodify(filepath, ":t:r")
      table.insert(files, { name = name, filepath = filepath, is_local = false })
    end
  end

  return files
end
```

The `scan_directory` function excludes README.md by design. This is appropriate for most artifact types (commands, skills, agents) where README.md is a directory index, not an artifact.

### Recommended Implementation Approach

**Option A (Recommended): Modify `create_docs_entries()` directly**

Replace the current scanning logic with direct README.md file detection. This avoids modifying `scan_directory` (which serves other artifact types correctly).

```lua
function M.create_docs_entries(config)
  local entries = {}
  local project_dir = vim.fn.getcwd()
  local global_dir = config and config.global_source_dir or scan.get_global_dir()
  local base_dir = config and config.base_dir or ".claude"

  -- Only show docs/README.md (not all .md files in docs/)
  local readme_path = nil
  local is_local = false
  local local_readme = project_dir .. "/" .. base_dir .. "/docs/README.md"
  local global_readme = global_dir .. "/" .. base_dir .. "/docs/README.md"

  if vim.fn.filereadable(local_readme) == 1 then
    readme_path = local_readme
    is_local = true
  elseif vim.fn.filereadable(global_readme) == 1 then
    readme_path = global_readme
    is_local = false
  end

  if readme_path then
    local description = metadata.parse_doc_description(readme_path)

    table.insert(entries, {
      display = helpers.format_display(
        is_local and "*" or " ",
        " " .. "└─",
        "README",
        description
      ),
      entry_type = "doc",
      name = "README",
      filepath = readme_path,
      is_local = is_local,
      ordinal = "zzzz_doc_README"
    })

    table.insert(entries, {
      is_heading = true,
      name = "~~~docs_heading",
      display = string.format("%-40s %s", "[Docs]", "Integration guides"),
      entry_type = "heading",
      ordinal = "docs",
      config = config,
    })
  end

  return entries
end
```

**Option B: Add filter parameter to scan_directory**

Add an `include_readme` or `filter_fn` parameter. This is more flexible but adds unnecessary complexity for a single use case.

**Option C: Add a docs_filter to config**

Add a `docs_filter` field to the shared config that limits which files appear. Over-engineered for this requirement.

**Recommendation**: Option A is the cleanest approach. It:
- Requires changes to only one function (`create_docs_entries` in entries.lua)
- Does not affect `scan_directory` behavior for other artifact types
- Works correctly for both Claude and OpenCode pickers via the `config.base_dir` parameter
- Is simple, explicit, and easy to understand

## Decisions

- Use Option A (direct modification of `create_docs_entries`) for simplicity
- Do not modify `scan_directory` since its README exclusion behavior is correct for all other artifact types
- Local README.md takes precedence over global (consistent with existing merge behavior)
- Use `"└─"` as the tree character since there is always exactly one entry
