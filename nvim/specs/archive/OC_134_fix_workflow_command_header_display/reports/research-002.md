# Research Report: Task #134 - Customizing OpenCode Header in Neovim

**Task**: OC_134 - Fix workflow command header not showing task number and name  
**Date**: 2026-03-04  
**Language**: meta  
**Focus**: Customizing OpenCode TUI header in Neovim to display task information from workflow commands

---

## Summary

The issue involves two distinct header systems that need coordination: (1) the OpenCode TUI session header showing "# New session - 2026..." and (2) the workflow command specification header that should display task information. The root cause is that while the workflow command specifications (research.md, implement.md, plan.md) include Step 3 "Display task header", this step is never executed because it's not implemented in the skill preflight stages. The TUI header is controlled by the OpenCode CLI itself, which has limited customization options, though there are active feature requests (#4539, #8436) for session title customization.

The best approach combines: (1) implementing the header display in skill preflight stages (building on research-001.md findings), and (2) using the existing WezTerm hook infrastructure to set custom user variables that can be displayed in the terminal title bar or status line.

---

## Findings

### 1. OpenCode TUI Header Architecture

The header showing "# New session - 2026..." is generated by the OpenCode CLI's TUI (Terminal User Interface) and is not directly customizable through the current API:

- **Session Title Generation**: OpenCode automatically generates session titles based on the initial prompt, but there are known issues (#6819, #7262) where titles get stuck on "New session - timestamp"
- **Customization Limitations**: There is no documented API to programmatically set the session title from external commands or hooks
- **Feature Requests**: Issue #4539 requests terminal window title customization, and #8436 requests editable session names with auto-generated titles

### 2. Existing Hook Infrastructure

The codebase already has a sophisticated hook system that detects workflow commands:

**Hook: wezterm-task-number.sh**  
- **Event**: UserPromptSubmit  
- **Function**: Parses `/research N`, `/plan N`, `/implement N`, `/revise N` commands and sets WezTerm user variable `TASK_NUMBER` via OSC 1337 escape sequences  
- **Code Location**: `.opencode/hooks/wezterm-task-number.sh:36-38`

```bash
if [[ "$PROMPT" =~ ^[[:space:]]*/?(research|plan|implement|revise)[[:space:]]+([0-9]+) ]]; then
    TASK_NUMBER="${BASH_REMATCH[2]}"
fi
```

The hook writes to the pane's TTY:
```bash
printf '\033]1337;SetUserVar=TASK_NUMBER=%s\007' "$TASK_VALUE" > "$PANE_TTY"
```

### 4. Neovim Plugin Integration

The user is using `opencode.nvim` (NickvanDyke variant) which provides:

- **Embedded TUI**: Uses OpenCode's native TUI within a Neovim terminal buffer  
- **Event System**: Forwards Server-Sent Events as `OpencodeEvent` autocmds  
- **Status Line**: Provides `require("opencode").statusline` component  
- **Configuration**: `lua/neotex/plugins/ai/opencode.lua` shows provider set to "snacks" with terminal integration

**Key Limitation**: The plugin doesn't expose an API to customize the session header displayed within the TUI.

### 5. WezTerm Integration Pattern

The existing WezTerm integration demonstrates best practices for terminal-level customization:

**OSC 1337 User Variables**:  
- WezTerm supports setting custom user variables via OSC escape sequences  
- These variables can be accessed in WezTerm's config (e.g., for tab titles)  
- Current implementation only sets `TASK_NUMBER`, but could be extended to set `TASK_NAME` and `ACTION`

**Hook Configuration** (settings.json lines 80-96):
```json
"UserPromptSubmit": [
  {
    "matcher": "*",
    "hooks": [
      {
        "type": "command",
        "command": "bash .opencode/hooks/wezterm-task-number.sh"
      }
    ]
  }
]
```

### 6. opencode.nvim Event System

The plugin forwards OpenCode events as Neovim autocmds:

```lua
vim.api.nvim_create_autocmd("User", {
  pattern = "OpencodeEvent:*",
  callback = function(args)
    local event = args.data.event
    -- event.type can be: "session.idle", "edit", "permission", etc.
  end,
})
```

This could be used to display task information in Neovim's status line or title bar, but not within the OpenCode TUI itself.

## Recommendations

### 1. Extend WezTerm Hook to Set Additional User Variables (Immediate Solution)

Modify `.opencode/hooks/wezterm-task-number.sh` to:
- Parse task number from command
- Look up task name from `specs/state.json`
- Set additional user variables: `TASK_NAME`, `TASK_ACTION`
- These can be displayed in WezTerm's tab bar or window title

**Implementation**:
```bash
# After extracting TASK_NUMBER, look up task name
TASK_NAME=$(jq -r --arg n "$TASK_NUMBER" 
  '.active_projects[] | select(.project_number == ($n | tonumber)) | .project_name' 
  specs/state.json 2>/dev/null || echo "")

# Set TASK_NAME variable
if [[ -n "$TASK_NAME" ]]; then
    NAME_VALUE=$(echo -n "$TASK_NAME" | base64 | tr -d '\n')
    printf '\033]1337;SetUserVar=TASK_NAME=%s\007' "$NAME_VALUE" > "$PANE_TTY"
fi
```

### 2. Implement Header Display in Skill Preflight (Architectural Solution)

Update skill preflight stages to display the task header before delegation:

**skill-researcher/SKILL.md**:
- Add header display in Preflight stage after validation
- Use plain text output (echo/print) to display the box header
- This appears in the chat output, not the TUI header

**skill-implementer/SKILL.md** & **skill-planner/SKILL.md**:
- Same pattern with appropriate action names

**Reference Implementation** (from research.md lines 42-53):
```
╔══════════════════════════════════════════════════════════╗
║  Task OC_N: <project_name>                               ║
║  Action: RESEARCHING                                     ║
╚══════════════════════════════════════════════════════════╝
```

### 3. Create Custom opencode.nvim Status Component (Neovim Integration)

Extend the Neovim configuration to display task information:

```lua
-- In lua/neotex/plugins/ai/opencode.lua
vim.api.nvim_create_autocmd("User", {
  pattern = "OpencodeEvent:session.start",
  callback = function()
    -- Read TASK_NUMBER from WezTerm user var or environment
    local task_num = vim.env.TASK_NUMBER
    if task_num then
      vim.g.current_opencode_task = task_num
      -- Update status line or winbar
    end
  end,
})
```

### 4. Use Session-Specific Environment Variables (Alternative)

Set environment variables when starting OpenCode that include task information:

```bash
# When launching opencode for a specific task
OPENCODE_SESSION_TITLE="Task OC_134: fix_workflow..." opencode
```

However, this requires launching separate OpenCode instances per task, which may not match the user's workflow.

### 5. Plugin-Based Solution (Future-Proof)

Create an OpenCode plugin (in `.opencode/plugins/`) that:
- Hooks into session lifecycle events
- Intercepts workflow command execution
- Modifies session metadata or display

This requires undocumented/unsupported APIs but provides the cleanest integration.
