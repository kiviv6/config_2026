## Key Findings

### 1. The Two-Layer Distinction Is the Central Documentation Problem

The most important strategic insight is that the existing documentation consistently conflates two fundamentally different layers:

- **Layer 1 (Neovim Lua)**: The extension loader lives in `lua/neotex/plugins/ai/shared/extensions/` — this is Neovim plugin code. It reads from `extensions/*/` and writes to `.claude/`. It has no dependency on `.claude/` files to function.
- **Layer 2 (Claude Code Markdown/JSON)**: The agent system lives in `.claude/` after extensions are loaded — these are files read by Claude Code CLI, not by Neovim.

Currently, `docs/architecture/extension-system.md` describes the extension mechanism well but does not explicitly call out that the loader is Neovim Lua code. Readers who are not familiar with the codebase may not realize the loader is a completely separate runtime environment from the agent system it manages. This conflation is the root cause of past confusion (including the "bootstrap impossibility" error documented in task 465's history).

### 2. Documentation Architecture Has Organic Fragmentation

Current documentation of the extension system is scattered across at least six locations:
- `docs/architecture/extension-system.md` — high-level architecture
- `context/guides/extension-development.md` — developer how-to
- `context/architecture/context-layers.md` — context layer boundaries
- `EXTENSION.md` (core) — core-specific extension description
- `docs/README.md` — navigation hub
- `CLAUDE.md` merge source — quick reference visible at agent session start

None of these explicitly documents the Lua extension loader as a system component. All of them describe the post-load `.claude/` state, but none describes the mechanism that creates that state. The README.md (`docs/README.md`) references `docs/architecture/extension-system.md` for extension architecture — this is the natural home for an expanded loader explanation.

### 3. The ROADMAP Has Adjacent Documentation Priorities That Align

The ROADMAP (`specs/ROADMAP.md`) already identifies three documentation priorities:

1. **Manifest-driven README generation** — a script that reads `manifest.json` and generates README starters. This is directly adjacent to task 468: once architecture is documented, a generation script would reduce future drift.
2. **CI enforcement of doc-lint** — automatically run `check-extension-docs.sh` on every push. This requires current docs to be accurate first (task 468 creates that baseline).
3. **`/review` surfacing of extension README drift** — requires the review command to understand what "correct" extension docs look like (task 468 defines that).

Task 468 is therefore a **prerequisite enabler** for all three ROADMAP items in the Documentation Infrastructure section.

### 4. The Core Extension Itself Lacks a README

Task 474 exists specifically to create `extensions/core/README.md`. The `check-extension-docs.sh` script fails because core has no README. This is directly adjacent to task 468's scope: both tasks are about documentation of the extension system, and creating the core README would benefit from the architecture clarity that task 468 produces. The two tasks should be aware of each other's work — specifically, task 474 should be scoped to reference the architectural understanding produced by 468.

### 5. Documentation Needs a "Source vs. Loaded" Vocabulary

A key confusion vector in existing docs: references to `.claude/agents/` or `.claude/context/` sometimes mean the extension source (`extensions/core/agents/`) and sometimes mean the loaded runtime destination. The docs need consistent vocabulary:

- **"Extension source"**: Files in `.claude/extensions/*/` — the canonical source of truth, tracked in git
- **"Loaded runtime"**: Files in `.claude/{agents,commands,context,...}/` — generated artifacts, populated only after loading

This distinction is absent from `project-overview.md`, `extension-development.md`, and the core `EXTENSION.md`.

### 6. Cross-Project Impact: This Sets the Pattern for Other Repos

The agent system is designed to be synced to other repositories (the "Load Core Agent System" sync mechanism). When another repository uses this system, their developers see only the `.claude/` runtime state — they never see the Neovim loader. Documentation that assumes familiarity with the Lua layer will be confusing to those users.

The `project-overview.md` (which lives in the core extension's context directory and is loaded by agents in every project) currently describes `.claude/` as containing the static structure, without explaining that this structure is generated by loading extensions. This is an important gap for cross-project users.

### 7. The Extension Loader Source Files Have No Agent-Side Documentation

The loader Lua module (`lua/neotex/plugins/ai/shared/extensions/`) has 8 files:
- `init.lua` — public API
- `loader.lua` — 628 lines, 12 public functions (copy_simple_files, copy_skill_dirs, copy_context_dirs, copy_scripts, copy_hooks, copy_systemd, copy_docs, copy_templates, copy_root_files, copy_data_dirs, check_conflicts, remove_installed_files)
- `manifest.lua`, `state.lua`, `merge.lua`, `picker.lua`, `verify.lua`, `config.lua`

There is no agent-accessible context file that documents what these functions do, what categories they handle, or what the loader's extension-to-deployment mapping looks like. Agents working on loader bugs (like task 470 — fix copy_context_dirs for root-level files) have to discover this by reading Lua source. A single concise reference document about the loader's function-to-category mapping would help.

## Recommended Approach

### Priority 1: Add an "Extension Loader" Section to `extension-system.md`

The highest-leverage documentation change is adding a new section to `docs/architecture/extension-system.md` that explicitly names and describes the Lua extension loader:

- What it is: Neovim plugin code, not Claude Code
- Where it lives: `lua/neotex/plugins/ai/shared/extensions/`
- What it does: reads manifests, calls copy functions per category, handles merge targets
- What categories exist and which functions handle them
- The complete lifecycle from "picker invoked" to "`.claude/` populated"

This is a targeted addition (~50-80 lines) to an existing document, not a new document.

### Priority 2: Update `project-overview.md` to Reflect Real Extension Architecture

The `project-overview.md` (in core's context, loaded by all agents) should be updated to show:
- `.claude/extensions/` as extension **source** directories (not just "Extension modules")
- That `.claude/{agents,commands,context,...}` are populated by loading extensions, not static
- The two-phase lifecycle: fresh state vs. loaded state

This is a small change with high agent-visibility impact (loaded every session).

### Priority 3: Establish "Source vs. Loaded" Vocabulary in `extension-development.md`

Add a brief "Terminology" section at the top of the extension development guide:
- Extension source (`extensions/{name}/`) vs. loaded runtime (`.claude/`)
- What "loading" means operationally

### Priority 4: Create a Loader Function Reference (New Document)

Create `.claude/extensions/core/context/guides/loader-reference.md` — a concise table of:
- Category → `manifest.provides` key → loader function → target path pattern
- Merge targets: CLAUDE.md, index.json, settings.json — how each works
- Copy semantics (file copy, directory copy, special handling)

This document helps with future tasks like 470 (loader bugs) and implementing new extension categories. It is agent-accessible (added to context index).

### Priority 5: Scope Documentation for Core README (Task 474 Coordination)

Task 474 should reference the architecture clarity from 468. The core README should explain:
- Core as the foundational extension (not a peer extension)
- What "foundational" means (no dependencies, loaded first, provides `merge_targets` that seed CLAUDE.md and index.json)
- Reference to the loader architecture doc for how loading works
